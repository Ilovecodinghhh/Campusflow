# CampusFlow 最终交付物设计

## 背景

现有 `产品方案V1.md` 已经形成了“智校空间运营助手”的完整概念，但范围偏大。最终交付物需要按照评估意见收敛为可展示、可试点、可验收的 B 端 AI 产品方案，而不是泛化的校园智能助手。

## 目标

将项目定位收敛为“面向高校教务/学工的校园空间预约与审批智能调度助手”，输出一组可用于课程作业、路演或内部评审的最终交付文档。

## 交付范围

交付包由五个 Markdown 文件组成：

1. `deliverables/00_最终交付物总览.md`
   - 说明项目一句话定位、交付物结构、MVP 边界、核心结论和推荐讲述顺序。
2. `deliverables/01_调研与需求分析.md`
   - 覆盖用户角色、访谈记录、用户旅程、痛点归因、需求优先级和 MVP 收敛理由。
3. `deliverables/02_PRD_校园空间预约与审批智能调度助手.md`
   - 覆盖产品目标、角色权限、核心流程、AI 工作流、数据模型、规则评分、异常兜底、指标体系和安全边界。
4. `deliverables/03_运营推广方案.md`
   - 覆盖试点学校选择、四阶段推进、冷启动策略、运营指标、采购切口和风险控制。
5. `deliverables/04_Demo方案与演示脚本.md`
   - 覆盖 Demo 信息架构、Mock 数据、三条演示链路、页面结构、技术实现和验收标准。

## 设计原则

- 聚焦 MVP：首版只证明“找空间”和“社团活动申请预审”两条闭环。
- 保留管理价值：BI 看板和周报作为展示层，用于体现 ROI，不作为首版重流程。
- 强调可控 AI：大模型负责意图识别、参数抽取、解释生成；结构化查询、规则过滤、预约写入由确定性系统完成。
- 补齐真实落地问题：明确数据可获得性、采购决策链、权限审计和人工复核。
- 保持 UTF-8 无 BOM；所有文档使用 Markdown。

## 资料依据

- 教育部 2025 年国家教育数字化战略行动部署：人工智能与教育变革、教育数字化转型。
- 教育部《“人工智能+教育”行动计划》相关解读：学校治理、资源分配、科学决策等应用方向。
- 教育行业标准《教育数据安全分类分级指南》：教育数据按安全属性、影响对象和影响程度分类分级。
- FineBI / FineChatBI 官方能力说明：自然语言问数、多轮追问、结果校验、数据解读、归因分析和趋势预测。

## 非目标

- 不写完整可运行产品代码。
- 不扩展到作业代写、心理咨询、校园生活全能问答等高风险泛化场景。
- 不假设已经打通所有高校系统接口；正式落地采用“Mock 数据演示、CSV 离线导入试点、API 对接扩展”的渐进路径。

## 验收标准

- 五份交付文档均存在，内容互相一致。
- MVP 边界、试点路径、验收指标清晰。
- 文档能支撑 8-12 分钟路演和后续 Demo 开发。
- 不出现空章节、未完成标记或明显自相矛盾内容。
